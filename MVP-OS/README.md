سند فلسفه و معماری MVP Engine (GPS پروژه)

هدف این سند: نقشه راه (GPS) پروژه است تا هر فرد یا AI با خواندن آن، فلسفه کاری، اجزای اصلی MVP Engine و نحوه تعامل آنها را به‌طور کامل درک کند. این سند قابل نگهداری و به‌روزرسانی در طول توسعه است.

---

۱. فلسفه کلی

تغییر بنیادین در نگرش توسعه:

· قبلاً:
    AI → Code
    (تولید کد مستقیم بر اساس خواسته)
· اکنون:
    Schema → UI → Code
    (ابتدا مدل داده (Schema) تعریف می‌شود، سپس UI و در نهایت کد تولید می‌شود)

این جهش به ما امکان می‌دهد تا به جای ساخت (Build) هر محصول از صفر، از کامپوننت‌ها و الگوهای قابل استفاده مجدد (Reuse) بهره ببریم. هدف نهایی، ایجاد Product Engine است که با تغییر یک فایل Schema، محصولات متفاوتی (کارت ویزیت، قراردادساز، رزومه‌ساز و ...) تولید کند.

---

۲. معماری لایه‌ای MVP Engine

معماری پروژه از چهار لایه اصلی و یک لایه انتزاعی برای داده تشکیل شده است:

```
Product Schema
       │
       ▼
  Wizard Engine   ←───  Libraries (SurveyJS, FormKit, ...)
       │
       ▼
   Providers      ←───  GoogleSheet, GoogleDrive, Auth, Payment, ...
       │
       ▼
   Dashboard      ←───  فقط Providers را صدا می‌زند
```

۲.۱. لایه Product Schema

· نقش: تعریف ساختار محصول به صورت یک فایل JSON مستقل از فریمورک و UI.
· مثال: card.schema.json

```json
{
  "steps": [
    {
      "id": "profile",
      "title": "اطلاعات اولیه",
      "fields": [
        { "type": "image", "name": "avatar" },
        { "type": "text", "name": "name", "required": true },
        { "type": "text", "name": "title" }
      ]
    }
  ]
}
```

· این فایل هیچ‌گونه وابستگی به React، Next یا هر کتابخانه خاصی ندارد.

۲.۲. لایه Wizard Engine

· نقش: خواندن Schema و رندر کردن فرم/مراحل (Wizard) بر اساس آن.
· روش پیاده‌سازی: از کتابخانه‌های موجود مانند SurveyJS، FormKit، TanStack Form یا React Hook Form استفاده می‌کنیم و فقط Renderer سفارشی خود را می‌سازیم.
· مسئولیت: تولید UI پویا و مدیریت وضعیت فرم.

۲.۳. لایه Providers

· نقش: تأمین سرویس‌های جانبی مانند ذخیره‌سازی، احراز هویت، پرداخت، تحلیل، آپلود و QR.
· همه Providerها به صورت پلاگین (قابل تعویض) طراحی می‌شوند.
· نمونه‌ها:
  · ذخیره‌سازی: GoogleSheetProvider (برای MVP)، بعداً PostgreSQL یا Supabase.
  · احراز هویت: Clerk، Better Auth، Auth.js.
  · پرداخت: ابتدا خالی، بعداً زرین‌پال.
  · تحلیل: ابتدا Google Sheet، بعداً Posthog.
  · آپلود: UploadThing یا GoogleDriveProvider.
  · QR: کتابخانه آماده.

۲.۴. لایه Dashboard

· نقش: نمایش اطلاعات و مدیریت محصولات با استفاده از Providerها.
· پیاده‌سازی: از کامپوننت‌های shadcn/ui، Tremor، Magic UI استفاده می‌کنیم.
· ویژگی: Dashboard اصلاً نمی‌داند داده از کدام Provider آمده است؛ فقط Providerها را صدا می‌زند.

۲.۵. لایه انتزاعی Repository Interface (جداسازی دیتا سورس)

برای جلوگیری از وابستگی مستقیم به یک ذخیره‌ساز خاص (مثلاً Google Sheet)، از الگوی Repository استفاده می‌کنیم:

```
Wizard
   │
   ▼
Application Service
   │
   ▼
Repository Interface
   │
   ┌───────────────┴───────────────┐
   ▼                               ▼
GoogleSheetProvider         PostgreSQLProvider
```

· در MVP فقط GoogleSheetProvider پیاده‌سازی می‌شود، اما هیچ جای دیگر برنامه (Wizard، Dashboard) از جزئیات آن مطلع نیست.
· این اصل برای تمام Providerها (ذخیره‌سازی، احراز هویت، پرداخت و تحلیل) اعمال می‌شود.

---

۳. جریان کاری محصولات

هر محصول جدید (کارت ویزیت، قراردادساز، رزومه‌ساز) فقط با تغییر Schema و در صورت نیاز تعویض یا افزودن Provider تولید می‌شود. Engine ثابت می‌ماند:

1. Schema تعریف می‌شود.
2. Wizard Engine فرم را تولید می‌کند.
3. کاربر داده را تکمیل می‌کند.
4. داده از طریق Repository Interface به Provider مناسب ارسال می‌شود.
5. Dashboard داده‌ها را بازیابی و نمایش می‌دهد.
6. خروجی نهایی (صفحه عمومی، پرینت، دانلود) منتشر می‌شود.

---

۴. بخش UI/UX Render

فعلاً خالی
در این نسخه از معماری، تصمیم مشخصی برای لایه رندر UI/UX گرفته نشده است. این بخش برای توسعه‌های آتی نگهداری می‌شود و می‌تواند شامل موتور تم‌سازی، سفارشی‌سازی بصری یا هر گونه بهبود تجربه کاربری باشد.

---

۵. نتیجه‌گیری نهایی

هدف واقعی پروژه، ساخت یک Product Engine است، نه صرفاً یک کارت ویزیت دیجیتال.
این Engine قادر است با یک Schema، کل زنجیره Wizard → Preview → Publish → Dashboard را برای هر محصولی اجرا کند.
این رویکرد با فلسفه Reuse به جای Build هم‌راستا بوده و از نظر فنی مقیاس‌پذیر، ماژولار و قابل نگهداری است.
